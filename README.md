# ТигрБанк - Модуль учета финансов

## О проекте
Консольное приложение для управления личными финансами по заказу Московского зоопарка. Моделирует доменную область финансового учета.

### Моделируемые сущности:
- **BankAccount** (id, name, balance) - банковские счета
- **Category** (id, type, name) - категории доходов/расходов
- **Operation** (id, type, bank_account_id, amount, date, description, category_id) - финансовые операции

## РЕАЛИЗОВАННЫЙ ФУНКЦИОНАЛ

### Основной (обязательный):
- ✔️ CRUD для счетов (создание, чтение, обновление имени, удаление)
- ✔️ CRUD для категорий (создание, чтение, обновление имени)
- ✔️ CRUD для операций (создание, чтение, удаление)
- ✔️ Автоматический пересчет баланса при создании/удалении операций

### Аналитика:
- ✔️ Разница доходов/расходов за период
- ✔️ Группировка доходов/расходов по категориям
- ✔️ Топ категорий расходов
- ✔️ Динамика по месяцам
- ✔️ Средний чек по категориям

### Импорт/Экспорт:
- ✔️ Экспорт всех данных в JSON
- ✔️ Импорт данных из JSON
- ✔️ DTO для безопасной сериализации

## ДЕМОНСТРАЦИЯ SOLID ПРИНЦИПОВ

| Принцип | Реализация |
|---------|------------|
| **S** (Single Responsibility) | Каждый класс имеет одну причину для изменения: AccountManager, CategoryManager, OperationManager, AnalyticsService |
| **O** (Open/Closed) | Интерфейсы репозиториев и DataExporter/DataImporter позволяют добавлять новые реализации |
| **L** (Liskov Substitution) | InMemory репозитории могут быть заменены любыми другими реализациями |
| **I** (Interface Segregation) | Разделение DataExporter и DataImporter вместо одного общего интерфейса |
| **D** (Dependency Inversion) | Зависимости направлены на интерфейсы, а не на конкретные классы |

## DI-КОНТЕЙНЕР (Google Guice)

```java
// TigerBankModule.java
bind(BankAccountRepository.class).to(InMemoryBankAccountRepository.class).in(Singleton.class);
bind(CategoryRepository.class).to(InMemoryCategoryRepository.class).in(Singleton.class);
bind(OperationRepository.class).to(InMemoryOperationRepository.class).in(Singleton.class);
bind(DataExporter.class).to(JsonDataHandler.class);
bind(DataImporter.class).to(JsonDataHandler.class);
```

Все зависимости внедряются через конструкторы с аннотацией `@Inject`.

## ПРОБЛЕМЫ ПРИ РАСШИРЕНИИ ФУНКЦИОНАЛА

1. **Валидация в менеджерах** - при добавлении новых правил придется менять существующие классы
2. **Отсутствие транзакционности** - возможна несогласованность данных при сбоях
3. **Прямая работа с репозиториями** - сложно добавить кэширование без изменения сервисов
4. **Консольный UI** - при добавлении GUI потребуется рефакторинг

## АРГУМЕНТЫ ЗА АБСТРАКЦИИ

| Абстракция | Что улучшила |
|------------|--------------|
| **Репозитории (интерфейсы)** | Можно менять способ хранения (in-memory → БД) без изменения бизнес-логики |
| **DataExporter/DataImporter** | Легко добавлять новые форматы (CSV, YAML) без изменения кода |
| **Менеджеры** | Инкапсулируют бизнес-правила, упрощают тестирование |
| **DI-контейнер** | Слабая связанность, централизованное управление зависимостями |

## ЕДИНЫЙ CODE STYLE

- **Отступы**: 4 пробела
- **Именование**: camelCase для методов/переменных, PascalCase для классов
- **Форматирование**: последовательное расположение аннотаций, модификаторов
- **Константы**: UPPER_SNAKE_CASE

## РАБОТА С ДАННЫМИ ИЗ ФАЙЛОВ

Реализован импорт/экспорт в **JSON** через Jackson:
- `DataContainer` - DTO для сериализации
- `JsonDataHandler` - имплементирует DataExporter/DataImporter
- Обработка ошибок при импорте
- Сохранение всех связей между сущностями
